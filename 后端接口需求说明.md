# 后端接口需求说明

## 问题说明

前端优化后，以下功能需要后端接口支持才能正常工作：

### 1. 分类匹配度计算接口

**问题**：当前分类匹配度是前端写死的（92%、90%等），需要根据用户真实偏好计算。

**需要接口**：
```
GET /recommend/category-matches/{userId}
```

**返回格式**：
```json
{
  "code": "0",
  "data": {
    "1": {  // 分类ID
      "match": 92.5,  // 匹配度（0-100）
      "trend": 0      // 趋势增长（百分比）
    },
    "2": {
      "match": 88.3,
      "trend": 15.2
    }
  }
}
```

**计算逻辑建议**：
- 根据用户历史订单和收藏的商品分类统计
- 匹配度 = (用户在该分类的商品数 / 用户总商品数) * 100
- 趋势 = 最近30天该分类的销量增长率

### 2. 推荐算法类型参数支持

**问题**：当前所有Tab切换都调用同一个接口，返回相同内容。

**需要修改接口**：
```
GET /recommend/user/{userId}/explanation
```

**新增参数**：
- `algorithmType` (可选): PERSONALIZED | COLLABORATIVE | CONTENT | TRENDING | SERENDIPITY
- `weights` (可选): JSON对象，包含各算法权重
  ```json
  {
    "collaborative": 60,
    "content": 30,
    "trending": 10
  }
  ```

**实现建议**：
- 根据 `algorithmType` 调用不同的推荐算法
- `weights` 参数用于混合推荐时的权重配置
- 如果未提供参数，使用默认的协同过滤算法

### 3. 算法设置接口

**问题**：算法权重设置只保存在前端，没有传递给后端。

**需要接口**（可选，如果前端传递参数到推荐接口，可以不需要单独接口）：
```
POST /recommend/algorithm-weights
Body: {
  "userId": 1,
  "weights": {
    "collaborative": 60,
    "content": 30,
    "trending": 10
  }
}
```

**或者**：在推荐接口中支持 `weights` 参数（推荐方式）

## 临时解决方案

在后端接口未实现前，前端已做以下处理：

1. **分类匹配度**：
   - 已添加 `fetchCategoryMatches()` 方法
   - 如果接口不存在，会使用默认值（向后兼容）

2. **算法切换**：
   - 未登录用户：根据算法类型获取不同排序的商品
   - 登录用户：传递 `algorithmType` 参数到推荐接口
   - 如果后端不支持参数，会降级到默认推荐

3. **算法设置**：
   - 权重保存在本地存储
   - 调用推荐接口时传递 `weights` 参数
   - 如果后端不支持，会使用默认权重

## 优先级建议

1. **高优先级**：推荐算法类型参数支持（影响核心功能）
2. **中优先级**：分类匹配度计算接口（提升用户体验）
3. **低优先级**：算法权重设置接口（如果推荐接口支持weights参数，可不需要）

## 测试建议

1. 测试不同算法类型返回不同的推荐结果
2. 测试算法权重调整后推荐结果的变化
3. 测试分类匹配度根据用户行为动态计算
